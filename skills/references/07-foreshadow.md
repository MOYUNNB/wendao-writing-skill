# 伏笔雷达 · 深度方法论

---

## 模块目标

管理伏笔的埋设与回收，防止烂尾。

## 5 层追问链（详细版）

### 第 1 层：埋什么伏笔？

**问题模板**：
- 你想埋什么伏笔？
- 读者什么时候该发现它？

**伏笔类型**：
| 类型 | 定义 | 例子 |
|------|------|------|
| **物品伏笔** | 关键物品出现 | 神秘戒指、传家宝 |
| **人物伏笔** | 关键人物出现 | 神秘少年、失踪的父亲 |
| **事件伏笔** | 关键事件发生 | 离奇死亡、神秘失踪 |
| **设定伏笔** | 关键设定揭示 | 主角的隐藏身世 |
| **关系伏笔** | 关系变化 | 反派的真实身份 |

### 第 2 层：回收情绪

**问题模板**：
- 这个伏笔的回收情绪是什么？
- 惊讶？感动？恍然大悟？
- 读者回收时应该有什么感受？

**情绪预设**：
- 惊讶型：作者没想到 → 妙
- 感动型：回顾时泪目 → 妙
- 恍然大悟型：原来如此 → 妙

**反模式**：伏笔回收时读者毫无感觉 → 失败。

### 第 3 层：未回收风险

**问题模板**：
- 如果这个伏笔迟迟没回收，读者会困惑吗？
- 需要备选方案吗？

**回收时机建议**：
- 短篇：埋设后 5-10 章内回收
- 中篇：10-30 章
- 长篇：30-100 章
- 超过 200 章未回收 → 失信风险

### 第 4 层：伏笔关联

**问题模板**：
- 这个伏笔和其他伏笔有没有关联？
- 能不能合并回收？

**伏笔网络**：
- 多个伏笔串联 → 集中回收（爽感加倍）
- 伏笔互为表里 → 一举两得

### 第 5 层：不回收怎么办？

**问题模板**：
- 如果这个伏笔最终不回收（有意为之），怎么让读者觉得「原来如此」而不是「你忘了」？

**不回收的处理**：
- 显式回收：作者直接说"这是伏笔，但已经不重要了"
- 隐式回收：在结局后用"真相"章节揭示（常见于无限流）
- 转化利用：伏笔变成另一件事的一部分

## 关键方法论

### 伏笔三要素

```
埋设位置 + 回收时机 + 回收情绪
```

缺一不可：
- 只有埋设无回收 → 烂尾
- 只有回收无情绪 → 无聊
- 只有情绪无时机 → 突兀

### 3/30/100 回收法则

| 章节范围 | 回收类型 | 情绪 |
|---------|---------|------|
| 3 章内 | 小伏笔 | 恍然大悟 |
| 30 章内 | 中伏笔 | 惊讶 |
| 100 章内 | 大伏笔 | 感动 |
| > 100 章 | 超长线 | 史诗感 |

### 6 种悬念工程

| 类型 | 定义 | 例子 |
|------|------|------|
| **信息差** | 读者知道但角色不知道 | 反派身份 |
| **认知差** | 角色以为 A，实际是 B | 误以为救星是敌人 |
| **时间差** | 现在 vs 过去的对比 | 主角回忆 |
| **空间差** | 这边 vs 那边的对比 | 战场两侧 |
| **身份差** | 表面身份 vs 真实身份 | 卧底 |
| **关系差** | 表面关系 vs 真实关系 | 假朋友真敌人 |

## 伏笔追踪表

**建议格式**：

```
| ID | 描述 | 埋设章节 | 预期回收 | 状态 | 关联 |
|----|------|---------|---------|------|------|
| F001 | 主角的金戒指 | 第 5 章 | 第 50 章 | planted | - |
| F002 | 神秘老者身份 | 第 8 章 | 第 100 章 | planted | F005 |
| F003 | 反派的过去 | 第 20 章 | 第 80 章 | planted | - |
```

**状态枚举**：
- planted（已埋设）
- half_revealed（部分揭示）
- resolved（已回收）
- abandoned（放弃）

---

## 反模式警告

| 反模式 | 触发信号 | 警告语 |
|--------|---------|--------|
| 挖坑不填 | 50 章前埋的伏笔无回收计划 | 你的读者会忘记，但作者忘了就是失信 |
| 回收过晚 | 完结前才回收关键伏笔 | 临时回收会让读者觉得仓促 |
| 回收过早 | 埋设 3 章就回收 | 失去悬念价值 |
| 强行反转 | 伏笔回收时强行反转 | 反转要符合已建立的逻辑 |
| 伏笔过密 | 5 章内埋 10 个伏笔 | 读者会怀疑作者炫技 |
| 伏笔过少 | 100 章只埋 1 个伏笔 | 长篇无伏笔，读者会觉得散 |

## SB 字段定义（foreshadowing）

```typescript
foreshadowing: {
  id: string;
  description: string;
  planted_at: string;        // 埋设章节
  planted_context: string;   // 埋设时的语境
  expected_payoff_at: string;// 预期回收章节
  payoff_emotion: string;    // 回收情绪
  status: 'planted' | 'half_revealed' | 'resolved' | 'abandoned';
  backup_plan: string;       // 备选方案（如果原计划失败）
  related_threads: string[]; // 关联伏笔 ID
}[]
```

---

## 完成标准

- 至少 1 条伏笔有 plant + payoff 计划
- 每条伏笔都有 description 和 expected_payoff_at
- 至少标注 1 种 payoff_emotion

---

## 推荐问题

1. "我想埋伏笔" → 进入 ⑦ 伏笔雷达
2. "我的伏笔太多了" → 进入伏笔关联（合并）
3. "我怕忘记伏笔" → 直接生成伏笔追踪表
4. "这条伏笔怎么回收？" → 进入第 2 层追问
5. "我不想回收某条伏笔" → 进入第 5 层追问（不回收的处理）

## 短篇视角：短篇无此模块（用"意象回声"替代）

> **触发条件**：length_type === "short"，用户提到"埋伏笔/伏笔/彩蛋"时。

**短篇的"伏笔"逻辑与长篇完全不同**——长篇伏笔是跨章节埋→收，短篇是同篇内埋→收。

**替代方案**：
- 意象回声系统 → 见 `references/short-beat-design.md` 的「意象回声系统」章节
- 2-3 个反复出现的意象，每次出现意义递进
- 海明威冰山理论：1/8 露出水面，7/8 藏水下

**短篇伏笔的 3 个特性**：
1. **同篇内**：开篇埋 → 结尾收，间隔可能只有 500 字
2. **数量少**：2-3 个意象（不是 10+ 个伏笔）
3. **意义翻转型**：不是"揭示"，是"赋予新含义"

**长篇伏笔 → 短篇意象回声** 的转换：
| 长篇 | 短篇 |
|------|------|
| 物品伏笔 | 反复出现的道具 |
| 人物伏笔 | 反复出现的小人物 |
| 事件伏笔 | 反复出现的场景 |
| 设定伏笔 | 反复出现的意象 |
| 关系伏笔 | 反复出现的对话/动作 |

**3/30/100 回收法则不适用**：
- 短篇所有伏笔必须在结尾前 1000 字内全部回收
- 短篇没有"超长线伏笔"（100 章 +）
- 短篇所有伏笔都是"近线"

**苏格拉底引导（短篇版）**：
- 问 1：你想重复出现的意象是什么？（开篇 + 中段 + 结尾）
- 问 2：第一次出现时是"普通细节"，结尾时新含义是什么？
- 问 3：意象数 ≤ 3 个（多了读者记不住）
- 问 4：开篇埋的意象，结尾前 1000 字内回收了吗？
